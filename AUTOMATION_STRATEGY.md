# Стратегия автоматизации

## Иерархия надёжности

### 1. REST API — максимальная надёжность
Для сервисов с API (Kaiten, Jira, Linear, Notion, GitHub) использовать HTTP-запросы напрямую.
Агент определяет *что* сделать, action выполняет через API.

```python
@controller.action("create_kaiten_child_card")
async def create_kaiten_child_card(parent_card_id: str, title: str) -> str:
    # один HTTP-запрос, никакого UI
```

### 2. Site-specific CDP skills — для сайтов без API
Детерминированные скрипты под конкретный сайт: навигация, ожидание селектора, клик, ввод.
Агент решает *что* делать, скилл знает *как* — не зависит от LLM-угадывания элементов.

### 3. browser-use с подробным промптом — для разовых задач
Подходит для непредсказуемых или одноразовых действий на любом сайте.
Ключ к надёжности — пошаговые инструкции с явным указанием UI-элементов:

```
1. Открой URL: https://...
2. Дождись загрузки панели карточки справа
3. Найди секцию "Дочерние карточки" внутри панели (не на доске)
4. Нажми кнопку "+" рядом с этой секцией
5. Введи название, сохрани
```

## Альтернативы browser-use

| Инструмент | Плюсы | Минусы |
|---|---|---|
| **Stagehand** | Более точный `act()`/`extract()`, построен на Playwright | TypeScript, платный хостинг |
| **Claude Computer Use** | Весь экран, не только браузер | Медленно, дорого |
| **Playwright напрямую** | Полный контроль, детерминировано | Нет LLM, только скрипты |

## Вывод

**browser-use** — хороший выбор для общего агента с непредсказуемыми задачами.
Для часто используемых сайтов — API или CDP-скиллы на порядок надёжнее и быстрее.

Правильный паттерн: **агент как оркестратор, скиллы как инструменты**.
